Alright team, let’s move on to part two of our arithmetic operators. The last slide covered the basics, but this slide? This is where the sneaky bugs live in your code. Today, we are looking at Prefix vs. Postfix placement, and the infamous integer division traps.
Let’s start on the left side. In the previous slide, we saw how the ++ operator adds 1 to a value. But where you place those plus signs changes the entire execution order. If you put them before the variable—which we call Prefix (++a)—Java says, 'Okay, I will increase this value right now, and then use the new number in the current expression.' However, if you put them after the variable—called Postfix (a++)—Java does the exact opposite. It says, 'I'm going to use the current value for this line of code right now, and I will increase it by 1 afterwards.' Also on the left, we revisit our integer division rule: dividing two integers truncates the decimal. We'll see how to fix that in a moment.
To really visualize how Prefix and Postfix differ, look at the execution flow in the middle section. Imagine we start with a base variable, x = 5. In the Prefix scenario (y = ++x), Step 1 happens immediately: x is bumped up to 6. In Step 2, that brand-new 6 is assigned to y. Both end up as 6. Now, contrast that with the Postfix scenario below it (z = x++). Step 1: we assign the current value of x to z. So z gets the 5. Only in Step 2 does x finally increase to 6. The end result? x is 6, but z is stuck at 5. It’s a subtle timing difference, but it can completely break a loop if you aren't paying attention.
Now, let's see these traps in action in the code block on the far right. The first two blocks just prove the theory we just discussed: b gets the updated 6 because of Prefix, while y gets the old 5 because of Postfix.
But I really want to draw your attention to section 3: The Integer Division Trap. Look at the badDiv variable. 5 / 2 gives us 2, because the decimals are thrown away. Now look right below it at trapDiv. We assigned 5 / 2 to a double—which is a decimal data type. You would expect the result to be 2.5, right? Wrong. It results in 2.0. Why? Because Java performed the integer division first, got a plain 2, and then saved that 2 as a decimal. The precision was already lost!
To get a proper, accurate decimal result, look at the very last example, goodDiv. At least one of the numbers in the operation must be explicitly written as a decimal, like 5.0. That forces Java to do floating-point math from the very beginning, finally giving us our correct result of 2.5. Keep an eye on your operator placements and your data types, and you'll avoid 90% of basic arithmetic bugs.
