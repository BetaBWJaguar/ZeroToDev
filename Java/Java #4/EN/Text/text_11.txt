Alright guys,welcome to part three of our arithmetic operators. Today, we are going to look at how to write cleaner and more professional code, and we'll also tackle one of the most confusing and favorite interview topics in Java: the mysterious dual identity of the plus (+) sign.
Let’s start on the left side with our first topic, Compound Assignment. As developers, we like being efficient—we don't want to write long, repetitive code. Instead of writing x = x + 5, Java gives us a fantastic shorthand: x += 5. This shortcut isn’t just for addition; it works for subtraction, multiplication, division, and modulo as well. But there is a very neat pro-tip hidden behind these compound operators: they automatically handle type casting behind the scenes. Normally, when you add different data types, Java might throw a compiler error, but when you use these shortcuts, Java quietly handles the conversion for you, and your code runs smoothly.
Right below that, we move on to our second topic: The Dual Role of the + Operator. The classic plus sign acts like a double agent in Java. If the values on both its left and right sides are numbers, Java performs standard math and adds them together. However, if even a single String (text) gets involved, that plus sign stops doing math entirely and acts like a powerful glue—which we call concatenation. And here is the golden rule: operations flow strictly from left to right. If the locomotive of the train—the very first operation—starts with a String, then all the numbers in the following cars turn into text one by one.
To really visualize this left-to-right flow rule, let's take a close look at the two scenarios in the middle section. In our first scenario, we see the expression 10 + 5 + "A". Java begins reading from left to right. In the first step, it encounters two numbers: 10 + 5. It says, "Great, I'm doing math!" and gets a result of 15. In the second step, it has to combine the number 15 with the string "A". The glue mode kicks in, and the result becomes "15A".
Now look at the second scenario, which is a classic exam question: "A" + 10 + 5. Java again starts reading from left to right. In the first step, it encounters the string "A" and the number 10. Since the string leads the way, it instantly switches to glue mode, resulting in the string "A10". In the second step, the string "A10" meets the number 5. Java glues them again, and the result is "A105". Many people automatically guess "A15" here, but no! Because of the left-to-right flow rule, the result is "A105".
Now, let's see how these rules come to life in the code block on the far right. In the first section, we assign the value 50 to our skor variable. By writing skor += 10, we bump the score up to 60, and then with the skor /= 2 shortcut, we divide the score to 30. You can see how clean and readable our code is.
The second section perfectly proves our string concatenation traps. We have variables x = 5 and y = 5. Look at the str1 line: the operation starts with numbers: x + y, which is 5 + 5. First, math is performed to get 10, which is then combined with the string to yield the output "10 Java".
However, in the str2 line, the operation starts with a string: "Java " + x + y. Since the head of the train is a string, the numbers 5 and 5 cannot be added; they are glued one by one, resulting in the surprising output "Java 55".
But what if we wanted to both start with a string and add the numbers? The solution is right there in the str3 line: the power of parentheses! When we write "Java " + (x + y), because the parentheses have operator precedence, Java first performs the 5 + 5 operation to get 10, then combines it with the string, giving us the clean, expected "Java 10" result. If you keep an eye on operator precedence and the left-to-right flow rule, you'll completely eliminate string concatenation bugs from your Java code.
