As we highlighted at the very top of our slide, the golden rule of bug-free coding in Java is mastering the critical difference between the "value copying" logic in primitive types and the "address sharing" logic in objects.
Our Developer Checklist on the left contains the essential security questions every developer should ask while writing code. First, remember that assignment in Java always flows from Right to Left—Java calculates the math on the right before writing it to the left. Second, that infamous bug of confusing = and == in if blocks; one performs an assignment and breaks your logic, while the other performs a comparison for a correct decision. Third, the "automatic type conversion" power behind compound operators like += and -= doesn’t just shorten your code; it saves you from tedious compile-time errors. Finally, we must never forget that primitive types are independent, but arrays or objects share a "memory controller."
So, how do we use these rules when we face a bug in a real project? Our Debugging Map in the middle section will guide you. Did you copy one array to another, and changing one broke the other? The reason is simple: = only copies the address, so both variables are linked to the same vault. Getting a compile error when adding numbers to a byte type? The solution is clear: instead of wasting time with x = x + 5, write x += 5 and let Java handle the type conversion behind the scenes. Or, if you want to get rid of a memory-heavy object, cut the link to the object by assigning null to the variable; this allows Java’s Garbage Collector (GC) to come in and wipe that RAM area clean.
Now, let’s crown all these rules with our code summary on the far right. In the first section, we see the power of compound assignment: after declaring byte b = 10;, writing b += 5; gets the job done. If we had written b = b + 5;, Java would have screamed "ERROR!". In the second section, we have that classic comparison trap: we use if (aktif == true) to perform the correct comparison. In the third section, we share array references: the moment we say diziB = diziA;, we are managing the exact same array; so the second we set diziB[0] = 9;, diziA[0] also becomes 9. In our fourth and final section, we have our cleanup and locking rules. We cut the link for A with diziA = null;. When we declare final int[] sabit, if we try to assign a new address like = new int[]{1}, Java throws an ERROR. But, with sabit[0] = 8;, it allows us to modify the data inside the array. This is exactly how net and consistent memory and assignment management works in Java. Every developer who turns these rules into second nature will write much safer code in the depths of Java.
