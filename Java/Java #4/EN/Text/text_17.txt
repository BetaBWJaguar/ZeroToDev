Welcome to part three of our assignment operators. In our previous slides, we learned the mechanics of basic, compound, and bitwise assignments. But today, we are tackling a topic that separates the beginners from the advanced developers: References and Memory. We are going to look under the hood to see how Java behaves differently when assigning primitive types versus complex types like Arrays and Objects. This is one of the most critical concepts in Java, and understanding it will save you from some of the most confusing, hair-pulling bugs you'll ever encounter in production.
Let's start on the left side with our first major concept: Value versus Reference Assignment. When dealing with primitive types like integers or doubles, Java performs a Value Copy. When you write int a = b;, Java copies the actual number into a completely separate box in memory. They become fully independent; changing one has absolutely zero effect on the other.
But look at what happens with complex types like Arrays or Objects. Here, Java performs a Reference Copy (or Shared Address). When you write int[] a = b;, Java does not copy the array. Instead, it copies the memory address—the pointer—to that array. Both variables now point to the exact same memory box. Consequently, modifying the data through one variable instantly changes the data for the other!
Below that, we have another fascinating rule: Assignment Evaluates to a Value. In Java, every time you assign a value, that operation actually produces a value—the very value that was assigned. For example, if you write System.out.println(x = 5);, Java does two things simultaneously: it assigns the number 5 to x, and it evaluates the entire expression as the number 5, printing it directly to the console. This hidden mechanic is exactly what makes chained assignments like a = b = 5 possible.
To visualize these two contrasting memory models, let’s look at the flow chart in the middle column. First, look at the Primitive Type flow. We start with int x = 10;. When we say int y = x;, y gets its own independent copy of 10. When we later change y = 20;, only y changes. x remains completely untouched at 10. They are fully independent.
Now contrast that with the Reference Type flow. We declare an array int[] a = {1, 2};. When we say int[] b = a;, b does not get a new array; it gets a copy of the address pointing to a. Both a and b are now holding controllers to the exact same array in memory. When we write b[0] = 99;, we are using b's controller to change the first element of that shared array. The result? BOTH CHANGE! If you check a[0], it has also magically become 99, because they are sharing the exact same memory space.
Let’s see these two behaviors run in our actual code block on the far right. In our first section, Primitive Assignment, we have int num1 = 10; and int num2 = num1;. This is our independent value copy. num2 is now its own separate 10. In section two, Reference Assignment, we have int[] arr1 = {1, 2, 3}; and int[] arr2 = arr1;. Here, the memory address is shared. When we write arr2[0] = 99;, we are mutating the shared array.
In section three, we demonstrate the return value of assignment: int x; System.out.println("x is: " + (x = 5));. This assigns 5 to x and instantly prints 'x is: 5' to the console.
Finally, when we print our results, we see that num1 remains completely unchanged at 10, proving our primitive independence. However, when we print arr1[0], we get 99—even though we made the change through arr2! This is the definitive proof of reference sharing in memory. Master this distinction, keep an eye on your memory pointers, and you will navigate complex object structures in Java with complete confidence. See you in the next slide!
