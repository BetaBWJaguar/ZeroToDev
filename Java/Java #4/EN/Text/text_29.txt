"Alright team, in our last slide, we looked at the Comparable interface and how it establishes a natural, default sorting order for our objects. But in the real world, requirements constantly change.
Imagine you have an e-commerce app with a list of products. Sometimes the user wants to sort them by price, sometimes by name, and sometimes by rating. Comparable falls short here because it only allows for a single, hardcoded sorting rule within the class. This brings us to the ultimate tool for changing sorting logic on the fly without modifying your original class code: the Comparator interface.
Let's look at the left panel to compare these two concepts. Think of Comparable as an internal rule. It is embedded directly within the class, providing a single, permanent natural order—like always sorting students by their ID. On the other hand, think of Comparator as an external referee that exists outside of your object. This means you can create ten different referees—one for age, one for price, one for name—and apply them to the exact same collection of objects whenever you need to.
To see how this 'engine of flexibility' works, let's follow the workflow in the middle section. When we need to sort a list dynamically based on user input, we simply call upon the appropriate referee. If the user selects 'Sort by Price', we apply PriceComp; if they choose 'Sort by Name', we use NameComp.
The workflow is straightforward: First, you define your Comparator rule. Second, you pass that rule directly into your sort method as a parameter. Finally, Java instantly overrides the natural order and sorts the list strictly according to your custom logic.
Let's see this flexibility in action in the right-hand code block. We start with a simple list of Strings containing "Zebra" and "Apple". Initially, we call Collections.sort(), passing only the list as a parameter. This relies on the natural Comparable order, so Java sorts the items alphabetically.
However, in the second section, we want a flexible sort. Instead of alphabetical order, we want to sort by word length. To do this, we create a custom Comparator called lenComp using a lambda expression that compares the lengths of the two strings. When we call Collections.sort() again—this time passing our new lenComp rule as the second parameter—Java disregards the alphabet and sorts the list strictly by word length.
This level of external control is exactly what makes Comparator the ultimate engine of flexibility in Java Collections."
