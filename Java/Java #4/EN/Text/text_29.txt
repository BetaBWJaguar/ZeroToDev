
Alright team, in our last slide we looked at the Comparable interface and how it sets a natural, default sorting order for our objects. But in the real world, requirements constantly change. Imagine you have an e-commerce app with a list of Products. Sometimes the user wants to sort them by price, sometimes by name, and sometimes by rating. Comparable gets stuck here because it only allows one hardcoded sorting rule inside the class. This brings us to the ultimate tool for changing sorting logic on the fly without touching your original class code: the Comparator interface. Let's look at the left panel to compare these two concepts. Think of Comparable as an internal rule; it is embedded directly within the class itself, providing a single, permanent natural order, like always sorting students by their Student ID. On the other hand, think of Comparator as an external referee. It exists outside of your object. This means you can create ten different referees—one for age, one for price, one for name—and apply them to the exact same object whenever you want. To see how this Engine of Flexibility works, let's trace the workflow in the middle section. If our goal is to sort a list differently based on user input, we simply call upon the right referee. If they click 'Sort by Price', we use our PriceComp. If they click 'Sort by Name', we use our NameComp. The workflow is incredibly simple: first, you write the Comparator rule. Second, you pass that specific rule as a parameter directly into your sort method. And third, Java instantly ignores the object's natural order and sorts the list strictly according to your custom rule. Let's see this powerful flexibility in action in our code block on the right. We start with a simple list of Strings containing "Zebra" and "Apple". In the first section, we call Collections.sort() with just the list as a parameter. This uses the natural Comparable order, so Java sorts them alphabetically. But in the second section, we want a flexible sort. We don't care about the alphabet; we want to sort by the length of the words. So, we create a custom Comparator called lenComp using a lambda expression that compares the lengths of the two strings. When we call Collections.sort() again, but this time pass our new lenComp rule as the second parameter, Java abandons the alphabet and sorts strictly by word length. This level of external intervention is what makes Comparator the absolute engine of flexibility in Java collections.
