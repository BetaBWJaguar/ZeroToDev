Let’s continue with our journey through Java's arithmetic operators. In the previous slides, we covered the general rules and some basic pitfalls. However, this slide focuses on the hidden, fine details that happen behind the scenes—under the hood. We are going to examine three crucial topics where developers often ask, "Why did this error happen?" and interviewers absolutely love to use to catch candidates off guard: the different reactions to division by zero, the magical way characters transform into numbers, and the implicit promotion of smaller data types.
Let’s start with our theoretical rules on the left. Our first topic is Zero Division. In classic mathematics, you cannot divide a number by zero, and Java knows this. But there is a huge distinction here: if you try to divide two integers (int) by zero, Java immediately halts the program and crashes by throwing an ArithmeticException. However, when it comes to decimals (double), Java does not crash! Instead, it safely returns special mathematical results like Infinity or NaN (Not a Number).
Our second detail is Char Arithmetic. In Java, char data types (like the letter 'A') are actually numbers behind the scenes; they store their corresponding ASCII or Unicode values. If you try to add two characters, or a character and a number, Java does not glue them together. It takes the character's underlying numeric value (for example, 65 for 'A') and adds them mathematically.
The third and most hidden trap is Implicit Type Promotion. When you perform an arithmetic operation on smaller data types in Java, such as byte or short, Java automatically promotes these variables to int during the operation. This is why you cannot add two byte numbers and directly assign the result to another byte variable; Java will throw a compiler error.
Let’s visualize how these scenarios happen at the memory level and in the execution flow using our flowchart in the middle section. First, let's look at the division by zero scenarios: 5 / 0 immediately leads to a 'HATA! (Crash)' because it is integer division. However, if you write it as a decimal like 5.0 / 0, the result is Infinity. If you perform an undefined division like 0.0 / 0, the result becomes NaN.
Right below that, we see the character operation flow: when we write 'A' + 1, Java takes the ASCII value of 'A', which is 65, adds 1 to it, and produces the result of 66. At the very bottom, we see the visual of our byte addition trap: byte x = 10 and byte y = 20 are defined. The moment we try to write byte z = x + y, Java says 'ERROR!'. This is because the underlying operation is executed as an int, and a larger data type (int) cannot be directly stuffed into a smaller box (byte).
Now, let's look at how these fine details come to life in the code block on the far right. In the first section, we see our division by zero tests. The line int hata = 5 / 0; is commented out because if it were active, it would crash our program. Right below it, by writing double sonsuz = 5.0 / 0;, we safely get the Infinity result, and by writing double belirsiz = 0.0 / 0.0;, we get NaN.
Let's move on to the second section: Char Arithmetic. We have defined char harf = 'A';. When we write int charToplami = harf + 5;, the ASCII value of harf (65) is added to 5, and the value of 70 is assigned to our charToplami variable. If we wanted to get the next character in sequence, we would write char siradakiHarf = (char) (harf + 1); to get the result of (char) 66, which is 'B'. Don't forget that we must perform manual casting here.
The final section proves that infamous Implicit Type Promotion trap. We have variables byte b1 = 10; and byte b2 = 20;. The line byte b3 = b1 + b2; is commented out because this line throws a compiler error. The correct approach is to capture the result as an int and write it as int toplam = b1 + b2;. Knowing these background rules and memory management in Java will completely protect you from unexpected crashes and compiler errors.
