Alright team, now we move on to a critical milestone in our operator journey: Relational (Comparison) Operators. In programming, defining data isn't enough; we often need to understand the relationship between values. Is one number greater than the other? Is a username correct? All these decisions are powered by these operators. As highlighted at the top of our slide, these operators compare two values and always return a logical decision—either true or false. They form the heartbeat of decision-making mechanisms like if-else blocks and loops in our code.
Let’s examine the Comparison Rules on the left. We are essentially dealing with two different worlds here: Primitives and Objects. When working with numeric data (int, double, char), life is simple; operators like == or != look directly at the value itself and never lead you astray. However, the real danger zone is the Reference/Object Rules. If you use == when comparing Objects—like Strings—Java does not look at the content; instead, it checks the "memory address" (the identity). This is why, if you want to verify if the text content of two objects is identical, you must forget the == operator and always reach for the .equals() method.
So, how does Java evaluate these comparison expressions under the hood? Let’s trace this in the Logic Evaluation Trace in the middle section. Imagine we have x = 10 and y = 20. When we write x != y, Java takes 10 and 20, compares them, and says, "Yes, they are different," returning true. When we check x >= y, it asks, "Is 10 greater than or equal to 20?" and returns false. But notice the trap in str1 == str2! Even if both hold the text "Java," because they exist in two separate memory boxes, this comparison returns false. This is the single most common mistake beginner developers fall into.
Now, let’s see how this logic comes to life in the code block on the right. In the first section, we define our variables a and b. The isEqual variable checks 10 == 20 and receives a false. Right below, we have an if block: if (a >= 10). Java evaluates this relationship; since 10 >= 10 is correct, it prints "A, 10 veya daha buyuk" (A is 10 or more) to the console.
In the third section, we test that infamous String trap. We have two separate String objects, s1 and s2. When we write (s1 == s2), our hatali (wrong) variable gets false because their memory addresses are different. But immediately below, when we use s1.equals(s2), Java looks at the text content and returns true to our dogru (correct) variable. It’s that simple! When you use relational operators with the correct data types and the right methods, you minimize the margin of error in your programs. See you in the next slide!
